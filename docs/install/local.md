# Локальная установка Metaflora Incubus v1

> **Статус релиза:** команды загрузки ниже подготовлены для готового v1, но пока не активированы. Они заработают после публикации проверенного GGUF, подписанного манифеста и контрольных сумм. Официальная ссылка на Hugging Face появится в README и release notes после этой проверки. Сейчас по этому адресу нельзя скачать готовую модель, и документ не выдаёт будущую ссылку за действующую.

Готовая установка занимает одну команду. Инсталлятор сам скачивает подходящий файл модели, проверяет подпись и SHA-256, ставит локальный рантайм, запускает API на `127.0.0.1` и добавляет Incubus в OpenCode. Окно установщика не открывается.

После установки используются два постоянных значения:

```text
API:      http://127.0.0.1:18991/v1
Model ID: metaflora-incubus-v1
```

`Model ID` в настройках клиента означает имя уже установленной Incubus. Ввод этого ID не скачивает вторую модель и не требует выбирать ещё какие-то веса.

## Самый короткий путь

### macOS и Linux

После выхода проверенного v1 открой Terminal и выполни:

```sh
curl -fsSL https://metaflora.ai/incubus/install | bash
```

Дождись строки `Metaflora Incubus v1 is ready`, перезапусти OpenCode и выбери:

```text
metaflora-incubus/metaflora-incubus-v1
```

На этом установка закончена. Локальный сервис будет подниматься вместе с пользовательской сессией и останавливаться при выходе из неё.

### Windows 10 и 11

Автоматическая Windows-установка не входит в v1: user-service и resource
preflight для неё не прошли чистую системную проверку, поэтому PowerShell
bootstrap не публикуется.

## Что именно делает команда

Инсталлятор выполняет действия в таком порядке:

1. Определяет операционную систему и архитектуру процессора.
2. Проверяет свободное место, оперативную память и доступное ускорение.
3. Загружает подписанный release manifest.
4. Проверяет подпись манифеста встроенным публичным ключом.
5. Скачивает подходящий GGUF и локальный рантайм во временную папку.
6. Сверяет размер и SHA-256 каждого файла.
7. Переносит проверенные файлы в пользовательскую папку Incubus.
8. Создаёт локальный сервис на `127.0.0.1:18991`.
9. Делает резервную копию конфигурации OpenCode и добавляет отдельный provider `metaflora-incubus`.
10. Запускает health check и откатывает установку, если API не отвечает.

Инсталлятор не меняет чужие provider-записи OpenCode. Повторный запуск безопасен: он проверит существующую установку и не создаст дубликаты.

## Если не хочется передавать скрипт прямо в shell

Короткая команда удобна, но перед запуском можно скачать и прочитать скрипт.

macOS и Linux:

```sh
curl -fSLo install-incubus.sh https://metaflora.ai/incubus/install
less install-incubus.sh
bash install-incubus.sh
```

## Требования к компьютеру

Основной релиз v1 планируется как GGUF размером около 3–5 ГБ. Итоговые цифры появятся рядом с файлом на Hugging Face после финальной квантизации и замеров.

| Конфигурация | Ожидаемый режим |
| --- | --- |
| 8 ГБ RAM | Экспериментальный профиль, короткий контекст, возможен своп. Для повседневной работы не рекомендуется. |
| 16 ГБ RAM | Основной профиль для чата, текста и небольших задач по коду. |
| 24–32 ГБ RAM | Более длинный контекст и меньше давления на память при работе с репозиториями. |
| Apple Silicon, 16 ГБ unified memory и больше | Локальный запуск с аппаратным ускорением. |
| NVIDIA, 8 ГБ VRAM и больше | Веса помещаются на GPU; точный запас зависит от контекста. |
| Только CPU | Работает медленнее, но отдельная видеокарта не обязательна. |

Для чистой установки оставь не меньше 12 ГБ свободного диска. Для обновления с возможностью отката лучше 18 ГБ: старая версия хранится до успешного health check новой.

Поддерживаемые платформы готового релиза будут перечислены в release notes. Инсталлятор остановится до скачивания модели, если для текущей системы нет проверенного рантайма.

### Профиль памяти

v1 выбирает параметры памяти автоматически по подписанному артефакту и ресурсам
компьютера. Ручное переключение профиля пока не входит в первый релиз. Точные
размеры контекста будут закреплены в release manifest после замеров.

## Проверка после установки

### Состояние сервиса

```sh
incubusctl status
```

Нормальный результат содержит версию, профиль, путь к GGUF, адрес API и строку `healthy`.

### Проверка API

macOS или Linux:

```sh
curl http://127.0.0.1:18991/v1/models
```

В ответе должен присутствовать ID:

```json
{
  "object": "list",
  "data": [
    {
      "id": "metaflora-incubus-v1",
      "object": "model",
      "owned_by": "local"
    }
  ]
}
```

Проверка генерации:

```sh
curl http://127.0.0.1:18991/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "metaflora-incubus-v1",
    "messages": [
      {"role": "user", "content": "Ответь одним словом: работает?"}
    ],
    "temperature": 0
  }'
```

## OpenCode

### Автоматическое подключение

Обычная установка находит пользовательский конфигурационный файл OpenCode, сохраняет его резервную копию и добавляет Incubus. Дальше нужны два действия:

1. Полностью перезапусти OpenCode.
2. Выбери `metaflora-incubus/metaflora-incubus-v1` в списке моделей.

API-ключ не нужен. Все запросы идут на локальный адрес, интернет после установки можно отключить.

### Если OpenCode был установлен позже

Повтори установочную команду. Скачивать 3–5 ГБ заново не придётся: инсталлятор сверит локальный GGUF и добавит только конфигурацию OpenCode.

Либо выполни:

```sh
incubusctl integrate opencode
```

### Ручное заполнение формы OpenCode

Если автоматическое обнаружение не сработало, заполни поля так:

| Поле | Значение |
| --- | --- |
| ID провайдера | `metaflora-incubus` |
| Отображаемое имя | `Metaflora Incubus v1` |
| Базовый URL | `http://127.0.0.1:18991/v1` |
| API ключ | оставить пустым |
| model-id | `metaflora-incubus-v1` |
| Отображаемое имя модели | `Metaflora Incubus v1` |
| Заголовки | не добавлять |

После сохранения модель будет называться:

```text
metaflora-incubus/metaflora-incubus-v1
```

Если интерфейс требует непустой API-ключ, впиши `local`. Сервер его не проверяет, поскольку принимает соединения только с этого компьютера.

### Ручная запись в конфигурации

Используй этот способ только после создания копии текущего файла:

```json
{
  "model": "metaflora-incubus/metaflora-incubus-v1",
  "provider": {
    "metaflora-incubus": {
      "name": "Metaflora Incubus v1",
      "npm": "@ai-sdk/openai-compatible",
      "options": {
        "baseURL": "http://127.0.0.1:18991/v1"
      },
      "models": {
        "metaflora-incubus-v1": {
          "name": "Metaflora Incubus v1"
        }
      }
    }
  }
}
```

Не заменяй весь существующий файл этим фрагментом. Добавь provider в имеющийся объект или используй `incubusctl integrate opencode`, чтобы сохранить остальные настройки.

## Ollama

Incubus может работать через собственный локальный API или через уже установленную Ollama. Вторая копия GGUF для регистрации не нужна.

### Установка с автоматической регистрацией

macOS и Linux:

```sh
curl -fsSL https://metaflora.ai/incubus/install | bash -s -- --register-ollama
```

Инсталлятор проверит наличие `ollama`, создаст локальную запись `metaflora-incubus-v1` и не будет скачивать модель из каталога Ollama.

### Ollama уже поставлена после Incubus

Повтори установку с флагом `--register-ollama` или выполни:

```sh
incubusctl integrate ollama
```

Проверка:

```sh
ollama list
ollama run metaflora-incubus-v1
```

Для программы, которая подключается именно к Ollama, используй:

```text
Base URL: http://127.0.0.1:11434
Model:    metaflora-incubus-v1
```

Если нужен OpenAI-compatible API Incubus, оставь `http://127.0.0.1:18991/v1`. Это два разных локальных адреса, но оба указывают на установленный GGUF.

## LM Studio

LM Studio умеет открывать локальные GGUF. После установки Incubus:

1. Выполни `incubusctl status` и скопируй путь из строки `Model file`.
2. Открой LM Studio.
3. Перейди в раздел локальных моделей.
4. Выбери импорт файла с диска.
5. Укажи файл `metaflora-incubus-v1-q4.gguf` по пути из первого шага.
6. Выбери импортированную модель и загрузи её.

LM Studio может скопировать файл в своё хранилище. Если в настройках есть вариант оставить файл на месте или создать ссылку, выбери его, чтобы не занимать ещё 3–5 ГБ.

Одновременно держать модель загруженной в LM Studio и в сервисе Incubus на компьютере с 16 ГБ памяти не стоит. Перед запуском в LM Studio останови сервис:

```sh
incubusctl stop
```

Вернуть обычный режим:

```sh
incubusctl start
```

## Continue

В Continue добавь OpenAI-compatible модель со следующими значениями:

```yaml
name: Metaflora Incubus v1
provider: openai
model: metaflora-incubus-v1
apiBase: http://127.0.0.1:18991/v1
apiKey: local
```

Точное расположение конфигурации зависит от версии Continue. Через интерфейс значения те же: provider `OpenAI`, Base URL `http://127.0.0.1:18991/v1`, model `metaflora-incubus-v1`, API key `local`.

После сохранения отправь короткий запрос. Ошибка `connection refused` означает, что локальный сервис не запущен; проверь `incubusctl status`.

## Любой OpenAI-compatible клиент

Для Zed, Roo Code, Cline, Aider, Open WebUI и других совместимых клиентов нужны одни и те же данные:

```text
Provider: OpenAI Compatible
Base URL: http://127.0.0.1:18991/v1
API key:  local        # только если поле обязательно
Model:    metaflora-incubus-v1
```

Некоторые клиенты просят URL без `/v1`, а затем дописывают его сами. Если запрос уходит на `/v1/v1/chat/completions` и получает 404, укажи `http://127.0.0.1:18991`.

Пример переменных окружения для CLI-инструмента:

macOS и Linux:

```sh
export OPENAI_BASE_URL=http://127.0.0.1:18991/v1
export OPENAI_API_KEY=local
export OPENAI_MODEL=metaflora-incubus-v1
```

Названия переменных у конкретного клиента могут отличаться. Адрес API и Model ID остаются теми же.

## Управление локальной моделью

### Запуск и остановка

```sh
incubusctl start
incubusctl stop
incubusctl restart
incubusctl status
```

### Логи

```sh
incubusctl logs
```

Логи не должны содержать содержимое запросов по умолчанию. Диагностический режим с телами запросов включай только осознанно и не отправляй такой лог публично без проверки.

### Обновление

```sh
incubusctl update
```

Обновление скачивается во временную папку, проходит проверку подписи и хеша, затем запускается на локальном health check. Старая версия удаляется только после успешной проверки новой.

### Откат

Установка и обновление сами возвращают предыдущую рабочую версию, если новый
runtime не проходит health check. Отдельной команды ручного отката в v1 нет.

### Полное удаление

```sh
incubusctl uninstall
```

Удаляются рантайм, GGUF, служба и provider Incubus. Регистрация Ollama остаётся:
тег мог быть заменён пользователем после установки, поэтому удалять его
автоматически небезопасно. При необходимости удали его вручную через
`ollama rm metaflora-incubus-v1` после проверки `ollama show`.

Если OpenCode-конфигурация Incubus была изменена вручную после установки, удаление не будет её молча перезаписывать. Инсталлятор покажет путь и предложит убрать запись самостоятельно.

## Установка без интернета на целевом компьютере

Офлайн-набор не входит в v1. Быстрая установка требует интернет для загрузки
подписанного манифеста, runtime и GGUF.

## Где лежат файлы

Точный путь показывает:

```sh
incubusctl status
```

По умолчанию используются пользовательские каталоги:

| Система | Каталог |
| --- | --- |
| macOS | `~/Library/Application Support/Metaflora Incubus` |
| Linux | `${XDG_DATA_HOME:-~/.local/share}/metaflora-incubus` |

Не перемещай GGUF вручную. Сервис и install state хранят его точный путь.
Перенос готовой установки на другой диск в v1 не поддерживается: удали Incubus
и установи заново после настройки системного каталога данных.

## Безопасность

Incubus по умолчанию принимает запросы только на `127.0.0.1`. Не меняй адрес на `0.0.0.0`, если не настроены аутентификация, firewall и шифрование. У локального API нет ключа, поэтому открытый порт даст любому устройству в сети доступ к модели и ресурсам компьютера.

Перед установкой проверяются три вещи:

- release manifest подписан ключом Metaflora, встроенным в `incubusctl`;
- SHA-256 скачанного файла совпадает со значением в подписанном манифесте;
- модель активируется только после локального health check.

Инсталлятор не загружает публичный ключ с того же сервера, где лежит модель. Иначе подмена сервера позволила бы заменить сразу файл и ключ.

Никогда не передавай в настройки локального клиента настоящий облачный API-ключ. Если форма не принимает пустое значение, используй строку `local`.

## Частые ошибки

### `curl: (22)` или `404`

Если v1 ещё не опубликован, это ожидаемо: endpoint установки пока не активирован. Проверь статус релиза в README репозитория. После публикации 404 означает, что используется устаревший адрес или сеть подменяет запрос.

### `incubusctl: command not found`

Закрой терминал и открой заново. Если команда по-прежнему не находится, выполни установщик повторно: он восстановит пользовательский PATH без повторной загрузки проверенного GGUF.

### `connection refused` на порту 18991

```sh
incubusctl status
incubusctl restart
incubusctl logs
```

Если порт занят другим процессом:

```sh
incubusctl doctor
```

Диагностика проверит файлы установки и health endpoint на `127.0.0.1:18991`.

### OpenCode не показывает модель

Сначала проверь API:

```sh
curl http://127.0.0.1:18991/v1/models
```

Если API отвечает, выполни:

```sh
incubusctl integrate opencode
```

Полностью закрой OpenCode, включая фоновый процесс, и запусти заново. В списке ищи `metaflora-incubus/metaflora-incubus-v1`.

### OpenCode просит API-ключ

Впиши `local`. Это техническая заглушка для формы, запрос останется на компьютере.

### Клиент получает 404

Проверь итоговый URL в его логах. Правильный endpoint для чата:

```text
http://127.0.0.1:18991/v1/chat/completions
```

Если клиент добавил `/v1` дважды, убери `/v1` из Base URL. Если он отправляет запрос в несовместимый endpoint, переключи provider на OpenAI-compatible chat completions.

### Модель отвечает слишком медленно

Проверь состояние runtime:

```sh
incubusctl status
incubusctl doctor
```

Закрой программы, занявшие GPU или большую часть RAM. Большой контекст замедляет
первый ответ и увеличивает расход памяти. Начни с короткого диалога, прежде чем
отправлять целый репозиторий.

### Система убивает процесс из-за памяти

Останови LM Studio и Ollama, если они держат тот же GGUF в памяти, затем перезапусти Incubus:

```sh
incubusctl restart
```

Один файл на диске может быть зарегистрирован в нескольких клиентах, но одновременная загрузка несколькими рантаймами расходует память несколько раз.

### Ollama не видит запись

```sh
ollama --version
incubusctl integrate ollama
ollama list
```

Если демон Ollama остановлен, сначала запусти его штатным способом, затем повтори интеграцию.

### Проверка хеша или подписи не прошла

Инсталлятор не активирует такой файл и удаляет временную загрузку. Повтори
установку из другой сети. Не отключай проверку подписи и не переименовывай
случайно скачанный GGUF в ожидаемое имя.

### Не хватает места для обновления

Проверь состояние через `incubusctl status`. Если свободно меньше 12 ГБ, удали
установку и повтори её после освобождения диска. Ручное удаление GGUF оставит
service state повреждённым.

### Нужна диагностика для issue

```sh
incubusctl doctor
```

Команда проверяет локальные файлы и health endpoint. Она не выводит тексты запросов.

## Что будет опубликовано вместе с v1

Команда быстрой установки станет действующей только когда релизный набор будет полным:

- проверенный GGUF для v1;
- результаты обязательных benchmark и regression-наборов;
- подписанный release manifest;
- SHA-256 для каждого артефакта;
- лицензия и обязательные юридические уведомления;
- карточка модели на Hugging Face;
- проверка чистой установки, OpenCode, Ollama, обновления и удаления на поддерживаемых системах.

До выполнения этих условий README не должен показывать фальшивую кнопку скачивания или утверждать, что модель уже доступна.
