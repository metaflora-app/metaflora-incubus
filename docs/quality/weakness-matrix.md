# Матрица слабых мест Incubus v1

Любая жалоба из Reddit, форума, GitHub или Hugging Face сначала считается
гипотезой. Она попадает в работу только вместе с точным окружением и минимальным
сценарием воспроизведения. Подтверждённый дефект становится regression-тестом и
остаётся в нём после исправления.

## P0: блокирует релиз

| Область | Что проверяем | Условие выпуска |
| --- | --- | --- |
| Формат tool calls | JSON Schema, вложенные аргументы, Unicode, parallel calls | Ни одного текстового или повреждённого вызова |
| Цепочка инструментов | Ошибка инструмента, повторный вызов, продолжение задачи | Агент восстанавливается и доводит задачу до проверяемого результата |
| Реальная работа агента | Изменение файлов, запуск тестов, поиск по репозиторию | Текст «готово» не засчитывается без фактического tool trace |
| Q5 | BF16, Q8, Q6, Q5 и финальный Q5 на одном наборе | Падение каждой ключевой метрики укладывается в зафиксированный бюджет |
| Галлюцинации | Вопросы без ответа, ложные предпосылки, RAG без нужного факта | Модель умеет признать отсутствие данных и исправляет ложную предпосылку |
| Необоснованные отказы | Разрешённые чувствительные запросы RU/EN | Доля отказов ниже порога, полезность не проседает |
| Benchmark provenance | Хеши весов, данных, шаблона, runtime и сырых ответов | Отчёт полный, подписан и относится к точному GGUF |

## P1: обязательно измеряется

| Область | Проверка |
| --- | --- |
| Длинный контекст | Парные задачи 8K/16K/32K/64K/128K, память ранних наблюдений, tool history |
| Повторения | Циклы, незакрытые reasoning-блоки, исчерпание token budget |
| Русский | Знания, редактура, длинный текст, типографика, код и выполнение инструкций |
| Английский | Те же категории на отдельном срезе |
| Apple Silicon | MTP выключен по умолчанию, пока не доказано ускорение без потери качества |
| Runtime parity | Одинаковые ответы и tool protocol в встроенном runtime и поддерживаемых клиентах |
| Скорость | TTFT, tokens/s, peak RAM, warm-prefix cache и время холодного запуска |

## Как закрывается найденная дыра

1. Фиксируем URL, дату, версию окружения и точный reproducer.
2. Отделяем дефект весов от ошибки runtime, parser, template или клиента.
3. Повторяем сценарий несколько раз на замороженных настройках.
4. Подтверждаем дефект независимым evaluator или проверяемым состоянием системы.
5. Добавляем пример в обучающий набор, если это законно и не загрязняет holdout.
6. Исправляем веса, шаблон или runtime в зависимости от причины.
7. Прогоняем BF16 и все релизные quant-варианты.
8. Оставляем сценарий в постоянном regression-наборе.

Фраза «исправлены все дыры» появится только вместе с реестром проверенных случаев
и сырыми результатами. Непроверенные сообщения остаются гипотезами и не влияют на
публичные заявления.
